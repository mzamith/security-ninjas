Broken Authentication & Session Management

With the lightning growth of the internet-driven application in today's world it's imperative to manage usernames and passwords. Not also because the exponentially growth in data, mobile and cloud tecnhologies, but also because the increment of security breaches happening every other day. Subsequently this topic has become more advanced to protect the data, systems and networks that everyone relies on.

Although all the efforts that are being made, this is nowhere near perfect, inasmuch as since 2004 Broken Authentication & Session Management has always stayed on the OWASP Top 10.

This type of flaws can be extremely serious not only because putting business at very high risk, lose confidential data is an option, but also because it can lead to openning of backdoors, to malicious attackers, on the entire company. Attackers, internal or external, can take advantage of this flaws in order to steal accounts and impersonate users. Once hijacked, as the attacker has the ability to do anything that the hijacked user could do, it can result in serious consequences affecting the company's viability/credibility.

Is my application vulnerable to this flaw?

Based on OWASP list your application is considered to be vulnerable when it complies with the following topics.

  • User authentication credentials aren't protected using hashing or encryption when storing. May lead to Sensitive Data Exposure
  • Credentials can be guessed or overwritten through weak account management functions (e.g., account creations, password changing, password recovering, weak session IDs)
  • Session IDs are exposed in the URL (e.g., URL rewriting)
  • Session IDs are vulnerable to session fixation attacks
  • Session IDs don't timeout, or user sessions or authentication tokens, particularly single sign-on (SSO) tokens, aren't properly invalidated during logout
  • Session IDs aren't rotated after succesful login
  • Session IDs, passwords, and other credentials are sent over unencrypted connection. May lead to Sensitive Data Exposure

How to prevent it

Following OWASP recommendations, should be available:

  • A single set of strong authentication and session management controls that should strive to:

    1. Meet all the context requirements defined in OWASP Application Security Verification Standard (ASVS) areas V2 (Authentication) and V3 (Session Management)
    2. Have a simple interface for developers. Consider using ESAPI Authenticator and User APIs
  • Strive to avoid XSS flaws that can be used to steal session IDs. See Cross-Site Scripting
