Insecure Direct Object References

Also called IDOR, this vulnerability refers to a reference to an internal implementation object, such as a file, directory or a database key is exposed without any access control. In such cases those references can be manipulated by an attacker intending to access unauthorized data.

Potential threads can come form an authorized user of the system that alters a parameter value that may lead to an object that he isn't allowed to access. This user may be authorized to use the system but not that specific object. If the application doesn't verify the authorization of the user to that specific object, it may result in a Insecure Direct Object reference.

Is my application vulnerable to this flaw?

According to OWASP the best way to find if an application is vulnerable is to verify that all object references have appropriate defenses

  • For direct references to restricted resources, does the application fail to verify the user is authorized to access the exact resource they have requested?
  • If the reference is an indirect reference, does the mapping to the direct reference fail to limit the values to those authorized for the user?

Security Ninjas Resolution

How to prevent

The only real solution, as of now, is to implement an access control system.

  • Use per user or session indirect object references. For example, use something less obvious that is harder to enumerate as a reference. E.g, a random string instead of an incrementing integer.
  • Check access. Each user of a direct object reference from an unstrusted source must include an access control check to ensure the user is authorized for the request object.
